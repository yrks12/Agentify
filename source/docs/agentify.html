Agentify · מדריך לימוד

איך הופכים אתר שלם לכלי AI

סוכן ה־AI קורא את האתר פעם אחת ומפיק SDK של פונקציות בפורמט JSON. משם והלאה — ה־AI משתמש באתר כאילו יש לו API, בלי לראות פיקסל אחד מהדף.

הבעיה

סוכני AI יודעים לקרוא ל־APIs בצורה מצוינת. הבעיה: רוב האתרים באינטרנט פשוט אין להם API. כדי להזמין משהו, למלא טופס, להוציא נתונים מטבלה — אדם פותח דפדפן, מסתכל בדף, ולוחץ. איך נותנים לסוכן AI לעשות בדיוק את אותו דבר?

הרעיון: שני שלבים נפרדים. בשלב הראשון (פעם אחת לאתר) סוכן חוקר את האתר ומפיק SDK — קובץ JSON עם רשימת פונקציות. בשלב השני (בכל שימוש), ה־LLM רואה רק את ה־SDK ובוחר פונקציה. מנוע דטרמיניסטי מבצע את המתכון בלי שום קריאת LLM נוספת.

זו הגישה שמייצרת SDK שנוצר אוטומטית, להבדיל מסוכן שמתבונן על האתר בכל הפעלה. היתרונות: מהירות, עלות נמוכה, החזרה דטרמיניסטית של אותו רצף. החיסרון: צריך מיפוי ראשוני; אם האתר משתנה מהותית — צריך למפות מחדש.

איך המיפוי עובד בפועל

  1. Crawler — סריקת האתר

    פותח דפדפן (Playwright), מגיע ל־URL ההתחלתי. בכל עמוד מזריק JS שמסמן כל אלמנט אינטראקטיבי בתכונה data-w2a-id, ומחזיר רשימה ממוספרת של רכיבים גלויים + קטע טקסט. עוקב אחרי עד 4 קישורי ניווט מאותו origin, ועדיפות לקישורים שנראים כמו עמודי פעולה (Contact, Search...).

  2. Proposer — LLM מציע כלים

    שולח את הסקר ל־gpt-5.4-mini עם system prompt שמבקש להציע 1–4 כלים. המודל מחזיר JSON עם שם (snake_case), תיאור, tool_type (action או extract), ו־JSON Schema של הפרמטרים. שמות הפרמטרים נגזרים משדות שהמודל ראה בעמוד (Name, Email, ...).

  3. Approver — אישור משתמש

    מציג את ההצעות בטרמינל. המשתמש יכול לאשר את הכול (Enter / --auto-approve), לבחור תתי־קבוצה לפי מספרי אינדקס, או לדחות. כלים שלא אושרו לא ייכנסו ל־SDK.

  4. Recorder — הקלטת מתכון לכל כלי

    עבור כל כלי שאושר:

    • כלי action (טופס, לחיצה): נוצר ערך־סנטינל לכל פרמטר — __W2A_NAME__, __W2A_EMAIL__... ה־RecordingBrowser (תת־מחלקה של Browser) רץ דרך הדפדפן ומבצע את הפעולות. כשהוא מזהה סנטינל בערך שנכתב — הוא מחליף אותו חזרה ל־ {{name}} במתכון. תפקיד+שם נשמרים כמו גם CSS גיבוי.
    • כלי extract (קריאת נתונים): שאילתה נפרדת ל־LLM שמייצרת ביטוי page.evaluate(...) שמחזיר JSON מובנה. המתכון הסופי הוא goto → wait → js_extract.

איך נראה מתכון בפועל

קובץ JSON שמכיל לכל כלי: שם, תיאור, סכמת פרמטרים (JSON Schema), ורצף פעולות אטומיות. {{name}} הוא תחליף לפרמטר שמוזרק בזמן ההפעלה. 

{
  "name": "submit_contact_form",
  "description": "שולח טופס יצירת קשר",
  "parameters": {
    "type": "object",
    "properties": {
      "name":  {"type": "string"},
      "email": {"type": "string"}
    },
    "required": ["name", "email"]
  },
  "steps": [
    {"op": "goto",   "url": "https://example.com/#contact"},
    {"op": "type",   "target": {"role": "textbox", "name": "Name *"},
                     "text": "{{name}}"},
    {"op": "type",   "target": {"role": "textbox", "name": "Email *"},
                     "text": "{{email}}"},
    {"op": "click",  "target": {"role": "button", "name": "Send"}},
    {"op": "wait",   "ms": 1500},
    {"op": "verify", "kind": "page_text_contains", "value": "thanks"}
  ]
}

אוצר המילים של ה־Engine

10 אופרציות, כולן דטרמיניסטיות, ללא LLM:

gotoניווט ל־URL
clickלחיצה על Target
typeמילוי שדה טקסט (תומך ב־{{param}})
selectבחירה ב־combobox
press_enterהקשת Enter
scrollגלילה (up / down / top / bottom)
waitהשהיה במילישניות
extractקריאת text / value / attribute מאלמנט
js_extractהרצת JS מותאם לחילוץ ערכים מורכבים
verifyאימות מצב; כשל ⇒ RecipeFailure

איך Target שורד שינויי DOM

כל יעד נשמר עם עד שלוש אסטרטגיות. ה־Engine מנסה אותן לפי סדר עדיפויות עד שאחת תופסת:

  1. role + name — דרך עץ הנגישות (ARIA). הכי יציב.
  2. css — נוצר במיפוי מ־id, name, data-testid או מיקום־בתוך־הורה.
  3. text — חיפוש טקסט גלוי.

הקבצים — מי עושה מה

11 קבצי פייתון, כל אחד אחראי על נושא אחד.

שכבת הדפדפן והתפיסה

browser.py

עוטף Playwright

פותח/סוגר דפדפן Chromium, מנווט, ומבצע פעולות (click, type_text, select_option, scroll, wait). כל פעולה מקבלת מזהה אלמנט ולא סלקטור גולמי.

ax_tree.py

הזרקת JS לבניית רשימת רכיבים

מזריק קוד JS לדף שמסמן כל אלמנט אינטראקטיבי גלוי בתכונה data-w2a-id, מסיק תפקיד (role) ושם נגישות, ומחזיר רשימה ממוספרת + קטע טקסט מהדף.

selectors.py

סלקטורים יציבים למתכונים

Target = {role, name, css, text}. בזמן הקלטה — לוכד את כל האסטרטגיות; בזמן הרצה — מנסה לפי סדר עדיפויות עד שמשהו תופס.

שכבת ה־SDK ומנוע הביצוע

recipe.py

Recipe + Engine (דטרמיניסטי)

Recipe = dataclass עם שם, תיאור, פרמטרים, וצעדים. Engine.execute(recipe, args, browser) מחליף {{param}} בכל שדה מחרוזת ומבצע כל op. כשל ⇒ RecipeFailure עם אינדקס הצעד והסיבה.

registry.py

קריאה וכתיבה של SDK־לאתר

load(slug) / save(registry) ל־ recipes/<slug>.tools.json. כולל to_openai_tools() שממיר לסכמת function calling של OpenAI.

שכבת המיפוי (Phase 1)

mapper.py

הצינור המלא של המיפוי

ארבעה שלבים בקובץ אחד: Crawler סורק עמודים → Proposer שולח ל־LLM ומקבל הצעות כלים → Approver מציג ב־CLI לאישור → Recorder מקליט מתכון לכל כלי שאושר.

recorder.py

דפדפן שמקליט את עצמו

תת־מחלקה של Browser. בכל פעולה — קודם לוכד את ה־Target ואת ה־text, ואז מבצע. ממיר ערכי־סנטינל (__W2A_NAME__) חזרה ל־{{name}}.

שכבת ה־LLM וה־CLI

llm.py

לקוח OpenAI + פרומפטים + סכמת כלים

קליינט OpenAI. משמש את ה־Proposer (להציע כלים), את ה־Recorder (כדי להעביר משימה סינתטית לדפדפן בזמן ההקלטה), ואת ה־Picker בזמן run-mapped (לבחור איזה כלי לקרוא).

cli.py

פקודות Typer

3 פקודות בלבד: map (מייצר SDK), call (קריאה ישירה לכלי בלי LLM), ו־run-mapped (LLM בוחר כלי ממשימה בשפה טבעית).

agent.py

לולאה פנימית של ה־Recorder

משמש את ה־Recorder כדי "להדגים" לדפדפן איך לבצע את הכלי שמוקלט. לא נחשף כפקודה ב־CLI — קוד תשתית.

memory.py

זיכרון פנימי

היסטוריה מתגלגלת של צעדים ובדיקת חזרתיות, בשימוש הלולאה הפנימית.

מה לא מטופל ואיך להרחיב

המערכת הנוכחית עובדת מצוין על טפסים של עמוד אחד ו־חילוץ נתונים מעמוד אחד. אחרי שהבנתם איך זה עובד — אלו חמש הנקודות שבהן זה נשבר, וכל אחת עם התיקון הקונקרטי שצריך:

1 אין הימשכות של session בין קריאות קטן

מה חסר כל agentify call פותח דפדפן חדש ונסגר בסוף. עוגיות לא שורדות בין קריאות. למה זה חשוב כל זרימה שדורשת התחברות חסומה. login(...) ואז view_cart() פשוט לא יעבוד — הקריאה השנייה היא דפדפן חדש, לא מחובר. איך לתקן להוסיף מחלקת Session ל־browser.py ששומרת BrowserContext חי בין קריאות, עם דגל --session <id> ב־CLI. run-mapped כבר רץ בקונטקסט אחד — רק call סובל מהבעיה.

2 אין אופ של איטרציה (אין pagination, אין for-each) בינוני

מה חסר מתכונים הם רצפים ישרים. אי אפשר להגיד "לכל סיפור ברשימה — תלחץ ותחלץ X". כדי לדפדף 5 עמודי HN צריך 5 מתכונים נפרדים. למה זה חשוב scraping, batch processing, "תראה לי את כל..." — כל משימה איטרטיבית. איך לתקן להוסיף op בשם for_each {items_target, sub_steps} ל־ recipe.py. ה־Engine מבצע page.locator(items_target).all() ומריץ את הצעדים הפנימיים עם משתנה {{_item}} פעיל.

3 אין הסתעפות (if_verify) בינוני

מה חסר verify רק עובר או נופל ב־RecipeFailure. אין "אם העגלה ריקה לך לקטלוג, אחרת לקופה". למה זה חשוב כל אתר עם מודלים, זרימות אופציונליות, הודעות אימות שצריך להגיב אליהן. ה־LLM יכול להחליט ברמה גבוהה, אבל לא הגיוני להעלות אליו כל מיני "האם יש פה כפתור Close או Cancel". איך לתקן להוסיף op בשם if_verify {kind, value, then: [...], else: [...]} ל־Engine ב־recipe.py. אם ה־verify עובר — להריץ את ה־then; אחרת את ה־else.

4 Crawler רדוד → הצעות כלים רדודות בינוני-גדול

מה חסר ה־Crawler ב־mapper.py רואה רק את עמוד הבית + עד ארבעה קישורי ניווט. עמודים מאחורי login, תוצאות חיפוש, wizard רב־שלבי או modals — בלתי נראים ל־Proposer. למה זה חשוב ה־Proposer הוא הצוואר הצר. בדיוק בגלל זה Wikipedia קיבל search_wikipedia אבל לא get_article_facts(query) — ה־Crawler מעולם לא ביקר במאמר. איך לתקן להפוך את ה־Crawler לסוכן בעצמו: למלא טפסים, ללחוץ על תפריטים, ולתעד מצב בכל שכבה. הופך לרקורסיבי ומצבי — וייתכן שצריך fixture של login.

5 קישור פרמטרים נכשל על פעולות לא־טקסט בינוני

מה חסר כשה־Recorder רואה type_text(id=4, text="__W2A_EMAIL__") הוא יודע להחליף ל־{{email}}. אבל בלחיצה על radio, checkbox, date-picker או file-upload — אין טקסט מוקלד, אין סנטינל, אין למה להיתלות. למה זה חשוב זה בדיוק מה שקרה ב־httpbin: pizza_size "נצרב" ל־"Small" במקום ל־{{pizza_size}}. כל טופס עם enum/select/radio מקבל פרמטרים חצי־פעילים. איך לתקן ה־Mapper צריך לעקוב איזה slot של placeholder "פעיל" כשפעולה לא־טקסטואלית נורית. להעביר ל־Agent את שמות הפרמטרים יחד עם ה־placeholders, ולבקש שיבחר אופציה שהטקסט הגלוי שלה מתאים. אחר־כך ה־Recorder מתאים role+name+value מול מפת ה־placeholders.

איזה כיוון לבחור?

החלטה ארכיטקטונית אחת כשמרחיבים: האם המתכון רצף שטוח או שפה קטנה?

מסלול X

מתכונים שטוחים, מורכבות אצל ה־LLM

להוסיף Session ולתקן את param binding. כל כלי נשאר פעולה אטומית קטנה (login, view_cart, checkout_step_1...). ה־LLM בזמן ריצה מרכיב ביניהם.

טוב: פשוט, צפוי, קל לתחזק. פחות טוב: יותר קריאות LLM.

מסלול Y

מתכונים כשפה קטנה

להוסיף for_each, if_verify, ו־ subcall ל־Engine. כלי אחד יכול להגיד "התחבר אם צריך, עבור N עמודים, חלץ הכל, החזר".

טוב: כלים חזקים, LLM רק בוחר ברמה גבוהה. פחות טוב: ה־Engine הופך למפרש קטן, יש יותר מה לבדוק וללמד.

סדר מעשי לקוד הזה: קודם (1) ו־(5) — קטנים, ערכם גבוה, פותחים 80% מהמקרים האמיתיים של מולטי־עמוד. אחר־כך (2) רק כשמשימה אמיתית דורשת pagination. (3) ו־(4) לדחות עד שתגיעו לקיר.

מסקנות חשובות שכדאי ללמד

  • אינטראקציה ≠ ידע על הדף. בזמן הריצה, ה־LLM פותר את המשימה בלי לראות אפילו פיקסל אחד מהאתר. רק שמות פונקציות וסכמת ארגומנטים.
  • המתכון הוא חוזה. JSON עם 10 אופ־קודים. אם האתר משתנה, ה־verify נכשל ב־RecipeFailure שצועק "תמפה מחדש" — לא דועך בשקט.
  • ה־AI מייצר את ה־SDK של עצמו. שורת ההדגמה: "הסוכן קורא את האתר פעם אחת, מפיק את ה־SDK שלו, ומשם משתמש באתר כמו ב־API."
  • שני סוגי כלים, שתי דרכי הקלטה. כלי action מוקלטים על־ידי הדגמה דרך דפדפן וערכי סנטינל; כלי extract מקבלים ביטוי JS שנוצר על־ידי LLM. שניהם מתאחדים לאותו פורמט מתכון.
